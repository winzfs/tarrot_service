# 00. 개발 원칙

이 문서는 `tarrot_service` 저장소에서 가장 먼저 읽어야 하는 기준 문서다. 모든 기획, 구현, 리팩터링, UI 결정은 이 문서의 원칙을 우선한다.

## 1. 프로젝트 정체성

`tarrot_service`는 단순한 타로 결과 생성기가 아니다.

이 프로젝트는 **신비로운 판타지 게임 속 점술관에 입장해, AI 점술사와 대화하며 타로 카드를 해석받는 웹 게임형 서비스**다.

핵심 키워드:

- 신비로움
- 게임 같은 몰입감
- 타로 카드 의식
- AI 점술사 NPC
- 카드 선택과 공개의 연출
- 부드럽지만 현실적인 조언

## 2. 사용자 경험 우선 원칙

사용자는 “결과 텍스트”를 보기 위해 들어오는 것이 아니다.

사용자는 다음 경험을 원한다.

1. 신비한 장소에 들어온 느낌
2. 질문을 의식처럼 작성하는 느낌
3. 직접 카드를 고르는 느낌
4. 카드가 뒤집히는 기대감
5. 점술사 NPC가 해석해주는 몰입감
6. 후속 질문을 통해 상담이 이어지는 느낌

따라서 기능을 만들 때는 항상 다음 질문을 먼저 한다.

> 이 기능은 사용자가 게임 속 점술관에 들어와 있다는 느낌을 더 강하게 만드는가?

## 3. 단순 HTML 앱 금지

이 프로젝트의 화면은 일반적인 폼 기반 웹앱처럼 만들지 않는다.

금지 방향:

- 일반 게시판 같은 UI
- 흰 배경 중심의 SaaS 폼 화면
- 버튼과 입력창만 있는 단순 페이지
- 텍스트 결과만 출력하는 챗봇 화면

권장 방향:

- Phaser 기반 게임 씬
- 카드 선택 연출
- 파티클, 빛, 안개, 마법진 효과
- RPG 대화창
- 점술사 NPC 말투
- 게임 메뉴 같은 버튼과 패널

## 4. AI는 기능이 아니라 캐릭터다

Gemma 계열 모델은 단순한 문장 생성 도구가 아니라 **점술사 NPC의 뇌**로 취급한다.

AI 응답은 항상 다음 성격을 가져야 한다.

- 신비롭다
- 부드럽다
- 사용자를 겁주지 않는다
- 확정적인 예언을 하지 않는다
- 현실적인 조언을 포함한다
- 게임 NPC처럼 몰입감 있는 말투를 유지한다

나쁜 예:

> 당신은 곧 이별할 것입니다.

좋은 예:

> 이 카드는 관계 안에 아직 말하지 못한 감정이 남아 있음을 비춥니다. 결론을 서두르기보다는, 먼저 작은 대화의 문을 여는 편이 좋겠습니다.

## 5. 타로는 예언이 아니라 자기성찰 도구다

서비스는 사용자의 불안감을 이용해서는 안 된다.

반드시 지킬 것:

- 죽음, 질병, 사고, 파산, 이별 등을 확정적으로 예언하지 않는다.
- 의료, 법률, 투자 판단을 단정하지 않는다.
- 상대방의 마음을 사실처럼 말하지 않는다.
- 사용자의 선택권을 빼앗지 않는다.
- 타로 결과는 가능성과 자기성찰의 언어로 표현한다.

기본 안내 문장:

> 카드는 미래를 고정하지 않습니다. 다만 지금의 마음과 상황을 다른 각도에서 바라보게 하는 거울입니다.

## 6. 프론트엔드는 게임, 백엔드는 의식의 해석자

역할을 분리한다.

### 프론트엔드

- Phaser 3 + TypeScript
- 게임 씬 구성
- 카드 선택, 뒤집기, 파티클, 대화창
- 사용자의 질문과 카드 선택 상태 관리

### Cloudflare Worker

- 정적 에셋 제공
- `/api/reading` 처리
- `/api/chat` 처리
- Workers AI 호출
- 프롬프트 구성과 안전한 응답 포맷 관리

### Workers AI

- Gemma 계열 모델 실행
- 타로 해석 생성
- 후속 대화 응답 생성

## 7. 모델은 교체 가능해야 한다

사용자는 Gemma2를 원하지만, Cloudflare Workers AI에서 사용 가능한 Gemma 모델명은 시점에 따라 바뀔 수 있다.

따라서 모델명은 코드에 하드코딩하지 않고 설정값으로 분리한다.

예시:

```ts
const model = env.AI_MODEL || "@cf/google/gemma-3-12b-it";
```

원칙:

- 모델명은 `wrangler.jsonc`의 `vars.AI_MODEL`에서 관리한다.
- 프롬프트, 응답 파서, UI는 특정 모델에 종속되지 않게 만든다.
- 나중에 Gemma2 모델이 명확히 제공되면 설정만 바꿔 적용한다.

## 8. 데이터와 연출을 분리한다

타로 카드 데이터, 스프레드 규칙, AI 프롬프트, 게임 씬은 서로 섞지 않는다.

권장 구조:

```txt
src/
  game/
    scenes/
    ui/
    effects/
  tarot/
    cards.ts
    spreads.ts
  api/
    client.ts
worker/
  index.ts
  prompts/
    reading.ts
    chat.ts
```

원칙:

- 카드 의미는 `tarot/cards.ts`
- 스프레드 규칙은 `tarot/spreads.ts`
- 화면 연출은 Phaser scene
- AI 프롬프트는 Worker 쪽
- API 호출은 프론트 API client

## 9. MVP는 작게, 분위기는 강하게

처음부터 모든 타로 기능을 넣지 않는다.

MVP에서 중요한 것은 기능 개수가 아니라 **첫 인상과 몰입감**이다.

MVP 범위:

- 시작 화면
- 질문 입력
- 3장 카드 선택
- 카드 뒤집기
- AI 리딩 결과
- 후속 대화 1회 이상

나중에 추가할 것:

- 오늘의 운세
- 연애운 스프레드
- 직업운 스프레드
- 카드 도감
- 리딩 기록
- 공유 이미지
- 사운드

## 10. 응답은 구조화한다

AI 응답은 가능하면 JSON 형태로 요청하고, UI는 그 구조를 기반으로 표시한다.

권장 응답 구조:

```json
{
  "title": "달빛 아래 열린 문",
  "summary": "지금은 불확실하지만 직감이 중요한 시기입니다.",
  "cards": [
    {
      "position": "과거",
      "name": "The Moon",
      "koreanName": "달",
      "reading": "최근 당신은 명확히 설명하기 어려운 감정 속에 있었습니다."
    }
  ],
  "advice": "서두르지 말고 확인 가능한 작은 신호부터 살펴보세요.",
  "npcLine": "별빛은 아직 희미하지만, 길은 사라지지 않았습니다."
}
```

원칙:

- UI가 파싱 가능한 형태를 우선한다.
- 실패 시 안전한 기본 메시지를 보여준다.
- 모델 출력이 깨져도 앱이 멈추면 안 된다.

## 11. 모바일을 처음부터 고려한다

이 서비스는 모바일에서 많이 사용될 가능성이 높다.

원칙:

- 세로 화면 기준으로 먼저 설계한다.
- 카드 3장은 모바일에서도 한눈에 보여야 한다.
- 대화창은 하단 고정형 RPG UI처럼 구성한다.
- 터치 영역은 충분히 크게 만든다.

## 12. 개발자가 지켜야 할 태도

이 프로젝트는 “타로 기능”을 만드는 것이 아니라 **작은 판타지 경험**을 만드는 것이다.

구현 중 판단이 필요하면 다음 우선순위를 따른다.

1. 사용자 몰입감
2. 안전하고 부드러운 AI 응답
3. 유지보수 가능한 구조
4. 성능
5. 기능 수

기능을 더하는 것보다 중요한 것은 사용자가 첫 10초 안에 이렇게 느끼게 하는 것이다.

> 뭔가 게임 속 신비한 장소에 들어온 것 같다.
