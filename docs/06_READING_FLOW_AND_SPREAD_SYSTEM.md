# 06. 리딩 흐름과 배열법 공통 시스템

이 문서는 `tarrot_service`에서 타로 배열법을 확장할 때 유지해야 할 **공통 연출 흐름, UI 구조, 해석 방식, 터치 규칙**을 정의한다.

목표는 카드 수만 늘리는 것이 아니다.

> 어떤 배열법을 추가하더라도, 사용자가 같은 점술관 안에서 같은 의식의 규칙을 경험하게 만든다.

## 1. 핵심 원칙

실제 타로 리딩은 카드 하나의 뜻만 읽지 않는다.

```txt
질문
  ↓
배열법
  ↓
카드 위치 의미
  ↓
카드 자체의 의미
  ↓
정방향/역방향
  ↓
카드 간 관계
  ↓
전체 흐름
  ↓
현실적인 조언
```

게임에서도 이 순서를 최대한 유지한다.

현재 구현은 과거/현재/미래 3장 배열을 기준으로 하지만, 앞으로 1장, 5장, 7장, 10장 배열을 추가할 때도 아래 공통 흐름을 재사용한다.

## 2. 현재 기본 흐름

현재 구현된 기본 흐름은 다음과 같다.

```txt
IntroScene
  ↓
QuestionScene
  ↓
질문 봉인 연출
  ↓
CardSelectScene
  ↓
카드 3장 공개
  ↓
ReadingScene
  ↓
제1장 카드 해석
  ↓
제2장 카드 해석
  ↓
제3장 카드 해석
  ↓
종장 종합 조언
  ↓
ChatScene
```

다른 배열법을 추가해도 이 큰 흐름은 유지한다.

변경되는 것은 다음뿐이다.

```txt
- 카드 수
- 카드 위치 이름
- 카드 위치 의미
- 카드 배치 형태
- 카드별 챕터 수
- 종장 종합 방식
```

## 3. 씬별 공통 역할

### IntroScene

역할:

- 사용자를 점술관으로 초대한다.
- 카드 뒷면과 VFX로 의식의 시작을 보여준다.
- 배열법과 무관하게 공통으로 사용한다.

배열법을 추가해도 기본적으로 변경하지 않는다.

### QuestionScene

역할:

- 질문을 받는다.
- 질문 카테고리를 받는다.
- 질문 봉인 연출을 보여준다.

확장 방향:

- 질문 카테고리에 따라 추천 배열법을 보여줄 수 있다.
- 연애 질문은 관계형 배열을 추천한다.
- 선택 질문은 선택 비교 배열을 추천한다.
- 자유 질문은 기본 3장 배열을 추천한다.

### CardSelectScene

역할:

- 배열법에 필요한 카드 수만큼 카드를 뽑는다.
- 카드 뒷면을 배열 위치에 배치한다.
- 사용자가 카드를 하나씩 공개한다.
- 모든 카드가 공개되면 리딩 단계로 이동한다.

배열법 확장 시 바뀌는 것:

```txt
- 카드 개수
- 카드 위치 좌표
- 카드 위치 라벨
- 공개 완료 조건
```

배열법 확장 시 유지할 것:

```txt
- 카드 공개의 의식적인 감각
- 중심축 180도 플립 느낌
- 발광/파동/파티클/VFX 계열
- 카드명 한글/영문 표시 원칙
```

### ReadingScene

역할:

- 카드별 해석을 챕터처럼 보여준다.
- 카드 위치 의미를 먼저 설명하고, 그 위치에 놓인 카드 의미를 질문과 연결한다.
- 마지막 종장에서 전체 흐름을 종합한다.

배열법 확장 시 바뀌는 것:

```txt
- 챕터 수
- 챕터 제목
- 위치 의미 문구
- 종장 종합 방식
```

배열법 확장 시 유지할 것:

```txt
- 챕터 진입 연출
- 카드 크게 보여주기
- 대화창 자동 표시
- 터치 잠금 규칙
- 종장 카드 융합/종합 조언 구조
```

### ChatScene

역할:

- 리딩 결과 이후 후속 질문을 받는다.
- 선택된 배열법, 카드, 위치 의미, 종합 조언 맥락을 유지한다.

배열법 확장 시 ChatScene 자체보다 API payload와 프롬프트 정보를 확장한다.

## 4. 공통 리딩 시퀀스

모든 배열법은 아래 시퀀스를 따른다.

### Step 1. 질문 봉인

목적:

- 사용자의 질문을 단순 입력값이 아니라 의식의 중심으로 만든다.

연출:

```txt
- 배경 어둡게 처리
- 질문 문장이 기도문처럼 표시
- 질문이 접히거나 봉인되는 연출
- VFX 발광/링/파티클
- 카드 선택 화면으로 전환
```

모든 배열법에서 동일하게 사용한다.

### Step 2. 카드 배치

목적:

- 배열법의 구조를 시각적으로 이해시키기.

연출:

```txt
- 카드 뒷면들이 배열 위치에 등장
- 각 카드 아래 또는 주변에 위치 라벨 표시
- 사용자가 하나씩 터치해서 공개
```

배열법별 차이:

```txt
1장: 중앙 1장
3장: 가로 3장
5장: 관계형/선택형에 맞춘 2열 또는 십자형
7장 이상: 전용 배치 필요
10장: 켈틱 크로스 전용 배치 필요
```

### Step 3. 카드 공개

목적:

- 실제 타로에서 카드를 한 장씩 뒤집어 확인하는 긴장감 구현.

공통 연출:

```txt
- 터치한 카드가 중심축 기준 180도 플립처럼 보임
- 공개 순간 빛 폭발
- 카드 앞면 이미지 표시
- 한글명/영문명 표시
- 공개 후 해당 카드 위치 라벨 유지
```

주의:

- 카드 공개는 즉시 결과를 모두 보여주는 것이 아니라, 사용자가 한 장씩 확인하는 의식이어야 한다.
- 카드가 많아져도 공개 방식은 동일해야 한다.

### Step 4. 카드별 챕터 해석

목적:

- 카드 의미를 단순 목록이 아니라 서사처럼 전달한다.

공통 구조:

```txt
챕터 제목 등장
  ↓
위치 의미/속삭임 표시
  ↓
카드 등장
  ↓
카드명 표시
  ↓
점술사 대화창 자동 표시
  ↓
3초 후 터치 가능
  ↓
다음 카드 챕터로 이동
```

예시:

```txt
제1장. 지나간 문
위치: 과거
의미: 현재 질문에 영향을 준 이전 흐름
카드: The Moon
해석: 과거의 불확실함이나 숨겨진 감정이 지금의 판단에 그림자를 드리운다.
```

### Step 5. 종장 종합 조언

목적:

- 카드들을 하나씩 해석한 뒤, 전체 흐름을 하나의 메시지로 묶는다.

공통 구조:

```txt
종장 제목 등장
  ↓
선택된 카드들이 빛으로 모이는 연출
  ↓
종합 조언 문장들이 한 줄씩 페이드인
  ↓
모든 문장 표시 후 3초 뒤 터치 가능
  ↓
후속 대화 화면으로 이동
```

종합 조언은 단순 요약이 아니다.

반드시 포함할 것:

```txt
- 질문에 대한 전체 흐름
- 반복되는 카드 성향
- 카드 간 연결
- 사용자가 지금 취할 수 있는 현실적 태도
- 확정 예언이 아닌 가능성의 방향
```

## 5. 터치 잠금 규칙

리딩 UX에서 가장 중요한 공통 규칙이다.

### 카드별 챕터

```txt
챕터 화면 진입
  → 터치 잠금
제목/카드 연출 진행
  → 터치 잠금 유지
대화창 자동 표시
  → 터치 잠금 유지
대화창 표시 후 3초 경과
  → 터치 가능
터치
  → 다음 챕터
```

### 종장

```txt
종장 화면 진입
  → 터치 잠금
카드 융합/본문 페이드인
  → 터치 잠금 유지
모든 문장 표시 후 3초 경과
  → 터치 가능
터치
  → ChatScene
```

목적:

- 사용자가 연출을 실수로 스킵하지 않게 한다.
- 카드가 하나의 장면처럼 읽히게 한다.
- 게임 챕터 연출 같은 호흡을 유지한다.

## 6. 배열법 데이터 구조 제안

배열법은 코드에서 데이터로 정의해야 한다.

권장 타입:

```ts
export type TarotSpread = {
  id: string;
  name: string;
  subtitle: string;
  description: string;
  recommendedFor: ReadingCategory[];
  cardsToDraw: number;
  positions: TarotSpreadPosition[];
};

export type TarotSpreadPosition = {
  id: string;
  label: string;
  chapterTitle: string;
  shortMeaning: string;
  promptMeaning: string;
  layoutHint?: {
    x: number;
    y: number;
  };
};
```

중요한 것은 `label`만 있으면 안 된다는 점이다.

AI 해석을 위해 반드시 필요한 정보:

```txt
- position id
- 화면 표시용 위치 이름
- 챕터 제목
- 짧은 의미
- 프롬프트용 상세 의미
```

## 7. 우선 추가할 배열법 후보

### 1장: 오늘의 한 장

용도:

- 빠른 조언
- 오늘의 흐름
- 매일 접속 유도

위치:

```txt
1. 오늘의 메시지
```

### 3장: 시간의 세 문

현재 기본 배열.

위치:

```txt
1. 과거
2. 현재
3. 미래
```

### 3장: 상황/장애물/조언

일반 고민에 적합.

위치:

```txt
1. 현재 상황
2. 막고 있는 것
3. 조언
```

### 5장: 관계의 거울

연애/인간관계에 적합.

위치:

```txt
1. 나의 마음
2. 상대의 마음
3. 관계의 현재
4. 막고 있는 것
5. 조언
```

### 5장: 선택의 갈림길

선택 고민에 적합.

위치:

```txt
1. 선택 A의 흐름
2. 선택 B의 흐름
3. 숨은 변수
4. 내가 진짜 원하는 것
5. 조언
```

### 5장: 내면의 나침반

심리/자기성찰 질문에 적합.

위치:

```txt
1. 겉으로 보이는 상황
2. 숨은 감정
3. 두려움
4. 필요한 태도
5. 조언
```

## 8. AI 해석 순서

AI는 각 카드를 다음 순서로 해석해야 한다.

```txt
1. 이 카드는 어떤 위치에 놓였는가?
2. 그 위치는 질문에서 무엇을 의미하는가?
3. 이 카드의 기본 상징은 무엇인가?
4. 정방향/역방향이면 그 에너지가 어떻게 드러나는가?
5. 사용자의 질문과 연결하면 어떤 의미인가?
6. 이전/다음 카드와 어떤 흐름을 만드는가?
7. 사용자가 지금 취할 수 있는 현실적 태도는 무엇인가?
```

현재 구현은 정방향/역방향을 아직 본격적으로 사용하지 않는다. 향후 확장 시 추가한다.

## 9. 카드 간 관계 해석

종장에서는 반드시 카드 간 관계를 본다.

확인할 것:

```txt
- 메이저 아르카나가 많은가?
- 같은 수트가 반복되는가?
- 숫자 흐름이 이어지는가?
- 특정 카드가 전체 분위기를 압도하는가?
- 과거에서 현재, 현재에서 미래로 흐름이 자연스러운가?
- 관계형 배열에서는 나와 상대의 카드가 충돌하는가, 보완하는가?
- 선택형 배열에서는 두 선택의 에너지가 어떻게 다른가?
```

예시:

```txt
Tower + Star
→ 무너짐 이후의 회복

Moon + Eight of Swords
→ 불안과 생각의 감옥

Death + Ace of Wands
→ 끝난 뒤 새롭게 시작되는 힘
```

## 10. UI 재사용 원칙

새 배열법을 만들 때 새 화면을 무조건 만들지 않는다.

우선 재사용할 것:

```txt
- 질문 봉인 연출
- 카드 뒷면 표시
- 카드 공개 플립
- 카드 앞면 표시
- 카드명 한글/영문 표시
- 챕터 제목 연출
- 카드 해석 대화창
- 종장 카드 융합 연출
- 전체 화면 터치 진행
```

배열법별로 새로 정의할 것:

```txt
- 카드 위치 데이터
- 카드 배치 좌표
- 챕터 제목 문구
- 위치 의미 문구
- AI prompt payload
```

## 11. 구현 우선순위

바로 확장한다면 다음 순서가 안정적이다.

```txt
1. spreads.ts를 데이터 중심으로 확장
2. 현재 과거/현재/미래도 spread 데이터에서 읽도록 변경
3. CardSelectScene이 spread.positions를 기반으로 카드 수와 위치를 결정하도록 변경
4. ReadingScene이 spread.positions를 기반으로 챕터 수와 제목을 결정하도록 변경
5. API payload에 spreadName, positionMeaning을 포함
6. AI 프롬프트가 위치 의미와 카드 간 관계를 해석하게 변경
7. 그 다음 5장 배열 추가
```

처음부터 켈틱 크로스 10장까지 확장하지 않는다.

이유:

- 모바일 배치가 복잡해진다.
- 카드 수가 많아지면 리딩 시간이 길어진다.
- AI 응답 품질 관리가 어려워진다.
- 현재 게임 리듬에는 1장, 3장, 5장 배열이 가장 적합하다.

## 12. 절대 바꾸지 말아야 할 기준

배열법 확장 중에도 아래 기준은 유지한다.

```txt
- 게임 화면은 viewport를 덮는다.
- 질문은 반드시 봉인 연출을 거친다.
- 카드는 사용자가 직접 공개한다.
- 카드 공개는 한 장씩 진행한다.
- 해석은 카드 위치 의미를 포함한다.
- 챕터/종장 연출 중 터치 스킵을 막는다.
- 대화창/본문 표시 후 3초 뒤 진행 가능하다.
- 최종 조언은 확정 예언이 아니라 가능성의 방향과 현실적 태도로 정리한다.
```
