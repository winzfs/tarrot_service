# AI 타로 리딩 프롬프트 / UI 플로우 가이드

이 문서는 질문 보조, 배열 추천, 카드 해석, 종장 해석, 카드 공개 연출, 결과 요약/공유 화면의 품질을 일정하게 유지하기 위한 기준이다.

## 현재 사용자 플로우

현재 리딩 플로우는 다음 순서로 동작한다.

```txt
질문 입력 → AI 질문 보조 → 배열 추천 → 질문 봉인 → 카드 선택/뒤집기 → 챕터 해석 → 종장 → 요약/공유
```

핵심 방향은 단순한 타로 결과 출력이 아니라, 사용자가 질문을 다듬고 카드 의식에 참여한 뒤 결과를 하나의 **별빛 기록 카드**로 남기는 **게임형 AI 점술 체험**이다.

## 공통 톤

- 점술사처럼 신비롭게 말하되, 사용자의 현실 판단을 방해하지 않는다.
- 확정 예언처럼 단정하지 않는다.
- 불안감을 과도하게 자극하지 않는다.
- `반드시`, `무조건`, `끝났다`, `절대` 같은 단정 표현을 피한다.
- 권장 표현: `~에 가까워 보여요`, `~을 비추고 있어요`, `지금은 ~을 확인해볼 때입니다`.
- 사용자의 질문에 없는 사실을 새로 만들지 않는다.
- 상대의 마음, 미래 결과, 제3자의 의도를 사실처럼 말하지 않는다.

## 1. 질문 보조

목표는 사용자의 질문을 길게 만드는 것이 아니라, **리딩 관점을 좁히는 것**이다.

입력 예시:

```txt
그 사람이 나를 어떻게 생각해?
```

좋은 보조 질문:

```txt
카드가 어느 방향을 더 비추길 원하나요?
```

좋은 선택지:

```json
[
  { "label": "상대의 마음", "appendText": "상대의 마음과 태도가 지금 어떤 흐름인지 알고 싶어." },
  { "label": "관계 흐름", "appendText": "앞으로 이 관계가 어떤 방향으로 이어질지 알고 싶어." },
  { "label": "내가 할 일", "appendText": "지금 내가 취하면 좋은 태도와 행동도 알고 싶어." }
]
```

규칙:

- 선택지는 3개만 제공한다.
- 사용자는 최대 2번까지만 선택해 질문을 다듬는다.
- 서로 다른 관점을 제시한다.
- appendText는 사용자의 원문 뒤에 붙어도 자연스러운 문장으로 쓴다.
- 사용자의 질문에 없는 사실을 만들지 않는다.
- 선택지는 `흐름`, `상대/관계`, `내가 할 일`, `막힌 이유`, `진짜 원하는 것`처럼 관점이 달라야 한다.

## 2. 배열 추천

배열 추천은 단순 키워드 매칭이 아니라 질문의 의도를 기준으로 한다.

판단 기준:

- 하루/가벼운 메시지/짧은 조언: `오늘의 한 장`
- 현재 문제와 해결책: `상황과 조언의 세 문`
- 관계/상대/연락/마음/재회: `관계의 거울`
- 둘 중 선택/이직/고백/계속할지 말지: `선택의 갈림길`
- 흐름/과거-현재-미래/전망: `시간의 세 문`

응답 필드 기준:

```json
{
  "spreadId": "relationship-mirror-five",
  "reason": "질문이 상대의 마음뿐 아니라 관계의 현재와 앞으로 취할 태도까지 함께 묻고 있어, 나·상대·관계·장애물·조언을 나누어 보는 배열이 적합합니다.",
  "refinedQuestion": "요즘 연락이 뜸한 상대와의 관계가 어떤 흐름인지, 상대의 태도와 내가 취할 태도를 함께 알고 싶어.",
  "detectedThemes": ["관계", "상대의 태도", "불안", "조언"]
}
```

규칙:

- reason은 1~2문장.
- refinedQuestion은 사용자의 의도를 보존한다.
- detectedThemes는 2~4개.
- 사용자가 직접 배열을 선택한 경우 AI 추천 이유를 덮어쓰지 않는다.
- 추천 배열이 도착하기 전에는 다른 배열 버튼과 질문 봉인 버튼을 열지 않는다.

### 배열 추천 UI 기준

배열 추천 화면에서는 상단 점술사 대화박스를 제거하고, 추천 본문 패널을 크게 보여준다.

추천 본문 패널은 다음 구성을 가진다.

```txt
추천 배열 제목
[작은 슬롯 미리보기]
읽힌 기운 : ...
다듬은 질문 : ...
위치 : ...
추천 이유 : ...
```

UI 규칙:

- `읽힌 기운 :`, `다듬은 질문 :`, `위치 :`, `추천 이유 :` 라벨은 포인트 컬러와 bold를 사용한다.
- 작은 슬롯 미리보기는 최대 5장까지 보여준다.
- 배열이 바뀌면 슬롯 라벨도 즉시 갱신한다.
- `다른 배열로 보기` 버튼은 AI 추천이 완료된 뒤 나타난다.
- 라벨/버튼 폰트는 작은 모바일에서도 읽히도록 기존보다 1px 크게 유지한다.

## 3. 카드별 챕터 해석

각 카드 해석은 반드시 다음 세 요소를 연결한다.

1. 사용자의 질문
2. 카드 위치 의미
3. 뽑힌 카드의 상징/키워드

나쁜 예:

```txt
이 카드는 새로운 시작을 의미합니다.
```

좋은 예:

```txt
관계의 현재 자리에 이 카드가 놓였기 때문에, 지금 두 사람 사이에는 완전히 끝난 흐름보다는 다시 방향을 정하려는 기운이 더 강하게 보입니다. 다만 서두르기보다는 상대의 반응을 확인하며 작은 신호부터 보는 편이 좋습니다.
```

### 카드별 해석 생성 규칙

카드별 `cards[].reading`은 AI가 처음부터 **2~3문장으로 생성**해야 한다.

프론트에서 뒤 문장을 잘라내는 것을 전제로 길게 생성하지 않는다. 3문장 안에 핵심이 들어가야 한다.

권장 구조:

```txt
1문장: 위치 의미와 카드의 핵심 결론
2문장: 카드 상징/키워드와 사용자 질문 연결
3문장: 사용자가 지금 확인하거나 해볼 수 있는 작은 현실 조언
```

규칙:

- 카드별 해석은 반드시 2~3문장.
- 4문장 이상 생성 금지.
- 1문장만으로 끝내지 않는다.
- 중요한 결론을 뒤쪽에 숨기지 않는다.
- 앞의 1~2문장만 읽어도 핵심이 전달되어야 한다.
- 위치 이름을 해석에 자연스럽게 반영한다.
- 상대의 마음은 사실처럼 단정하지 않는다.
- 현실적으로 취할 수 있는 태도를 한 가지 포함한다.
- 프론트는 안전장치로 최대 3문장까지만 표시한다.

## 4. 종장 해석

종장은 챕터별 카드 해석과 다르게 취급한다. 카드별 해석을 다시 반복하지 않고 전체 흐름을 요약한다.

중요: **챕터 해석은 2~3문장으로 압축하지만, 종장 advice 규칙은 별도로 유지한다.**

배열별 종장 문장 수:

- 1장 배열: 2문장
- 3장 배열: 3~4문장
- 5장 배열: 4~5문장

프론트 표시 안전장치:

- 종장은 최대 4문장까지만 화면에 표시한다.
- AI가 5문장을 생성할 수 있어도, 화면 안정성을 위해 표시 단계에서는 4문장으로 제한한다.

구성:

1. 결론
2. 이유
3. 조언
4. 필요 시 흐름 정리 또는 다음 행동

예시:

```txt
지금 이 관계는 끝을 말하기보다 거리를 조절하며 다시 균형을 찾는 흐름에 가깝습니다. 카드들은 상대의 마음보다 두 사람 사이의 표현 방식과 타이밍이 더 중요한 변수라고 보여줍니다. 당장은 큰 결정을 내리기보다, 부담 없는 방식으로 작은 확인을 해보는 것이 좋습니다.
```

규칙:

- 카드별 해석을 그대로 반복하지 않는다.
- 전체 카드가 만드는 흐름을 하나로 묶는다.
- 결론을 먼저 말한다.
- 같은 의미를 반복하지 않는다.
- 점술사가 종장에서 직접 건네는 대사처럼 작성한다.

## 5. 카드 선택 / 공개 연출 기준

카드 선택 화면은 사용자가 의식에 참여하는 느낌을 주는 장면이다.

현재 기준:

- 3장 배열은 카드 사이 간격을 넓게 두고 크게 보여준다.
- 5장 배열은 윗줄과 아랫줄 간격을 충분히 띄운다.
- 선택 배열의 `선택 A`, `선택 B` 위치 라벨은 카드 선택 화면에서 숨긴다.
- 카드를 뒤집으면 해당 카드가 중앙으로 이동하며 확대되고, 카드명도 함께 보여준다.
- 확대 카드 표시 시간은 충분히 길게 유지한다.
- 모든 카드가 열리면 본문 안내 문구가 아니라 중앙 팝업으로 `모든 봉인이 열렸습니다`를 보여준다.
- 이 팝업은 등장할 때 배경을 살짝 어둡게 만들고, 사라질 때 배경도 함께 풀린다.

## 6. 챕터 화면 레이아웃 기준

챕터 화면은 `제목 → 카드 → 해설` 리듬을 유지한다.

규칙:

- 챕터 제목은 약간 더 오래 보여준다.
- 카드와 해설은 설명 길이에 따라 아래로 보정한다.
- 챕터 해석문은 최대 3문장 표시.
- 문장 단위로 줄바꿈해 어색한 중간 끊김을 줄인다.
- 한국어 줄바꿈은 `keep-all` / `line-break: strict` 기준으로 처리한다.
- 해설 박스는 normal / long / extra-long 길이 분기를 사용한다.

## 7. 종장 화면 레이아웃 / 연출 기준

종장은 챕터 화면과 별도 레이아웃으로 다룬다.

구성:

```txt
상단: 카드 합성 연출 + 종장 제목
중앙/하단: 종장 해설 문장들
하단: 요약 화면 안내
```

연출 규칙:

- 카드 합쳐지는 연출은 챕터 전환보다 느리게 보여준다.
- 종장 해설은 카드 합성 연출이 어느 정도 끝난 뒤 나타난다.
- 종장 해설 문장은 천천히 등장한다.
- 긴 종장 해설은 `long-advice` 레이아웃으로 분기한다.
- 제목과 해설이 겹치지 않도록 종장 전용 레이아웃을 유지한다.
- 종장 마지막 터치 후 `다시 묻는 방`이 아니라 `별빛의 기록` 요약/공유 화면으로 이동한다.

## 8. 요약 / 공유 화면 기준

리딩의 마지막 화면은 후속 질문 입력이 아니라, 결과를 저장하고 공유할 수 있는 카드형 요약 화면이다.

요약 화면 구성:

```txt
별빛의 기록
- 배열 이름 / 카드 수
- 리딩 제목
- 내 질문
- 선택 카드 목록
- 전체 요약
- 종장 해석
- 공유하기 / 텍스트 복사 / 새 질문 버튼
```

UI 규칙:

- 요약은 하나의 카드 형태로 보여준다.
- 선택 카드 목록에는 위치, 카드 한글명/영문명, 짧은 해석 요약을 함께 표시한다.
- 카드 이미지가 있으면 작은 썸네일을 표시한다.
- 공유하기 버튼은 Web Share API를 우선 사용한다.
- Web Share API가 없거나 취소되면 텍스트 복사로 fallback한다.
- 텍스트 복사 버튼을 별도로 제공한다.
- 새 질문 버튼은 `QuestionScene`으로 돌아간다.
- 기존 `ChatScene`은 남아 있을 수 있지만 기본 플로우에서는 진입하지 않는다.

공유 텍스트 구성:

```txt
별빛의 기록

질문: ...

배열: ...

선택 카드:
- 위치: 카드명

요약: ...

종장: ...
```

## 9. UI 안정성을 위한 응답 길이 제한

- question assist guidance: 45자 이내 권장
- followUpQuestion: 40자 이내 권장
- assist option label: 8자 이내 권장
- spread reason: 90자 이내 권장
- card reading: 2~3문장, 180자 이내 권장
- finale advice: 배열별 문장 수 유지, 180~240자 이내 권장
- summary card item reading: 96자 내외로 축약 표시

## 10. 안전한 리딩 원칙

- 건강, 법률, 금융 문제는 전문가 상담을 대체하지 않는다고 부드럽게 안내한다.
- 자해, 폭력, 스토킹, 통제 행동을 부추기지 않는다.
- 상대를 조종하거나 감시하라는 조언을 하지 않는다.
- 사용자의 불안을 이용해 반복 결제를 유도하는 문구를 쓰지 않는다.
- 타로는 자기성찰과 관점 정리를 돕는 도구라는 관점을 유지한다.
